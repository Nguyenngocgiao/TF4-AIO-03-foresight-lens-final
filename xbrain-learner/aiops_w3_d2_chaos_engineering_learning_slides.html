01

Định nghĩa và tư duy đúng

AIOps W3 · D2

Chaos Engineering là gì?

“Thực nghiệm có chủ đích inject fault vào distributed system để tìm weakness trước khi fault tự xảy ra trong production.”

Fault injection là hành động gây lỗi: kill pod, delay network, fill disk...

Chaos Engineering là quy trình khoa học quanh hành động đó: giả thuyết → đo steady-state → giới hạn blast radius → inject → đo kết quả → rollback → học.

Không có hypothesis và measurement thì đó chỉ là “phá thử”, chưa phải chaos engineering.
AIOps W3 · D2

So sánh với test khác

Thực hànhMục tiêuKhi chạyĐiểm khác với Chaos
Unit testVerify code đúng specCI/CDKhông test interaction giữa service thật
Load testVerify chịu tải dự kiếnPre-launch, periodicTăng traffic, không chủ đích gây failure mode
Penetration testTìm security vulnerabilityQuarterlyFocus vào security, không focus reliability cascade
Chaos engineeringTìm reliability weakness do interactionContinuously / prod-likeGây fault thật, đo user-visible steady-state
AIOps W3 · D2

Tại sao AIOps cần Chaos?

Không có chaos

  • Detector chỉ được test trên dataset “đẹp”.
  • Correlator tưởng gom đúng nhưng chưa gặp alert flood thật.
  • RCA có vẻ hợp lý nhưng chưa bị cascade đánh lừa.
  • LLM output nghe thuyết phục nhưng chưa validate bằng ground truth.

Có chaos

  • Ta biết fault nào pipeline bắt được, fault nào miss.
  • Có MTTD: mất bao lâu mới detect.
  • Có RCA accuracy: chọn đúng culprit hay nhầm victim.
  • Có gap analysis để improve W1/W2 pipeline.
Chaos ở W3-D2 là “exam” cho pipeline W1 Detection + W2 Correlation/RCA.

5 nguyên tắc cốt lõi

AIOps W3 · D2

Nguyên tắc 1 — Build hypothesis around steady-state

Sai

“System should still work.”

Đúng

“Trong 60 giây partition payment-svc, order_success_rate ≥ 99.5% và checkout_p99 ≤ 800ms; nếu không, retry/fallback chưa đạt.”

Steady-state là trạng thái “user vẫn ổn”, không phải “container vẫn chạy”. Nên đo bằng SLI hoặc synthetic probe.
AIOps W3 · D2

Nguyên tắc 2 — Vary real-world events

🌐Network

Delay, packet loss, bandwidth throttle, partition, DNS fail.

🔥Resource

CPU stress, memory fill, disk fill, I/O saturation.

💥Application

Kill pod, pause process, HTTP 500, slow response.

⏱️State

Clock skew, bad config, cache poisoning.

Không chọn fault ngẫu nhiên. Chọn fault mà production thực sự có thể gặp: dependency timeout, DB memory pressure, retry storm, DNS slow.

Region1 region / cell
Global GameDayall regions, controlled
Không skip stage. Stage trước fail thì stop → fix → retry. “Run in production” không có nghĩa là nhảy thẳng vào 100% traffic.

Experiment design template — 5 field bắt buộc

Không có đủ 5 field này thì experiment khó chấm, khó rollback, khó học được gì.

experiment:
  name: "Payment service network partition under load"
  hypothesis: |
    Steady-state: order_success_rate >= 99.5%, checkout_p99 <= 800ms.
    Khi payment-svc bị partition, retry/fallback sẽ recover trong < 30s.
  blast_radius:
    target: "1 instance of payment-svc"
    traffic: "10% canary traffic"
    duration: "60 seconds"
  rollback:
    automatic: true
    trigger_when: "order_success_rate < 90% OR checkout_p99 > 3s"
    method: "iptables flush, restart sidecar"
  measurement:
    metrics: [order_success_rate, checkout_p99, payment_retry_count]
    capture_window: "t-5min to t+10min"
  ground_truth:
    root_service: payment-svc
    expected_fault: network_partition

Application faults — khi process bị tác động trực tiếp

Pod/container kill

Kiểm tra availability, restart policy, readiness, alert route.

Pause / SIGSTOP

Process không crash nhưng không xử lý request. Dễ lừa healthcheck đơn giản.

HTTP 500 inject

Test error-rate detector và retry storm.

Slow response

Test timeout, latency SLO, bulkhead.

Exception injection

JVM/bytecode/test-specific. Mạnh nhưng dễ intrusive.

Bad deploy simulation

Test canary rollback và RCA “bad_deploy”.

State faults — lỗi khó thấy nhất

Clock skew

JWT/cert/TLS/request signing fail. Service vẫn “healthy” nhưng auth fail hàng loạt.

time skew +60s

Config corruption

Sai feature flag, sai endpoint, sai timeout. Often root cause nằm ở config push chứ không phải code.

replace config file

Cache poisoning

Redis trả data sai/stale. Request vẫn 200 nên internal metric có thể OK.

Scheduler, rate limiter, token expiry behave bất thường.

Synthetic probe script

Steady-state đề xuất: ≥ 99% pass trong window 60s. Before inject pass, during inject đo impact, after rollback phải recover.

#!/usr/bin/env bash
# synthetic_probe.sh — log pass/fail mỗi 5s
ENDPOINT="${1:-http://localhost:8080/checkout/health}"
LOG="${2:-probe.log}"
while true; do
  ts=$(date -u +%s)
  start=$(date +%s%N)
  code=$(curl -s -o /dev/null -w "%{http_code}" --max-time 2 "$ENDPOINT")
  end=$(date +%s%N)
  latency_ms=$(( (end - start) / 1000000 ))
  if [[ "$code" == "200" && "$latency_ms" -lt 500 ]]; then
    echo "$ts pass $latency_ms" >> "$LOG"
  else
    echo "$ts fail $code $latency_ms" >> "$LOG"
  fi
  sleep 5
done

06

Pipeline failure modes cần chaos để lộ ra

AIOps W3 · D2

Failure mode 1 — Detector miss vì anomaly chìm dưới noise floor

Triệu chứng

Metric có lỗi thật nhưng baseline vốn đã dao động mạnh. 3σ band quá rộng nên real anomaly không vượt ngưỡng.

Counter

Dùng p99 thay vì mean; baseline theo segment peak/off-peak; anomaly trên residual hoặc SLO burn.

AIOps W3 · D2

Failure mode 2 — Correlator false positive

Vấn đề

Hai fault độc lập xảy ra trong cùng 5 phút: deploy bug A + network blip B. Nếu chỉ dùng time-window, chúng bị gom thành một incident.

Counter

Topology-aware correlation: phải cùng time-window và gần nhau trên dependency graph. Orphan alert vẫn ra cluster riêng.

AIOps W3 · D2

Failure mode 3 — RCA pick service ồn nhất

payment-svc
root
→
checkout-svc
retry 10×
→
edge-lb
timeout
Naive RCA: “checkout nhiều alert nhất” → pick checkout. Đúng: payment là root, checkout chỉ là victim do retry storm.
Counter: graph traversal + temporal lag + evidence link.
AIOps W3 · D2

Failure mode 4 — LLM hallucination confidence cao

Vấn đề

LLM trả root cause nghe hợp lý, confidence 0.9, nhưng service không có trong cluster hoặc không có metric evidence.

Guardrail

  • root_cause ∈ cluster.services
  • class ∈ enum
  • confidence ∈ [0,1]
  • actions non-empty
  • phải có evidence link
AIOps W3 · D2

Failure mode 5 — Monitoring dependency loop

Monitored service
depends on
Consul / DB / Queue
also used by
Monitoring stack
fails →
Silent blackout
Nếu AIOps stack phụ thuộc vào chính service đang monitor, khi dependency chết thì telemetry cũng chết. Chaos phải test meta-monitoring.
Counter: observability stack riêng, blackbox probe độc lập, alert khi telemetry ingestion lag.

def build_inject_cmd(exp: dict) -> list[str]:

    if fault == "network_loss":
        return ["pumba", "netem", "--duration", "60s", "loss", "--percent", "30", target]

Câu hỏi vấn đáp thường gặp

Chaos khác load test ở đâu?

Load test tăng tải để xem capacity. Chaos inject fault thật để xem system degrade/recover và pipeline có phát hiện/RCA đúng không.

Vì sao cần synthetic probe?

Vì metric nội bộ có thể nói OK trong khi user bị lỗi do DNS/LB/TLS/ingress. Probe ngoài cluster đo user-visible steady-state.

Blast radius là gì?

Là phạm vi thiệt hại tối đa cho phép. Bắt đầu nhỏ: 1 pod, 1% traffic, duration ngắn, có rollback.

Khi RCA sai thì ghi gì?

Ghi symptom, experiment nào, likely cause nằm ở detector/correlator/RCA, đề xuất fix cụ thể. Không sửa số để pass.

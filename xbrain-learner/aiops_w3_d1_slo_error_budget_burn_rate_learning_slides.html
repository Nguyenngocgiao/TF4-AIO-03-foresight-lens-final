01
Mental Model

SLO không phải monitoring đẹp hơn.
Nó là cách ra quyết định.

Alert đúng nghĩa phải trả lời: user đang đau chưa, còn bao nhiêu budget, có cần page ngay không?

Why this matters

Vấn đề của alert kiểu cũ

Threshold tĩnh như CPU > 80% hoặc error_rate > 5% thường không biết user có thật sự bị ảnh hưởng hay không.

False positive

CPU 85%, latency vẫn 120ms, user OK. Nhưng pager vẫn kêu.

False negative

CPU 30%, deadlock, p99 = 5s, user rất đau. Nhưng CPU alert im.

SLO-based alert

Chỉ page khi service đang đốt error budget nhanh đến mức cần hành động.

Core vocabulary

SLI, SLO, SLA — ba tầng khác nhau

SLIĐo cái gì?
→
SLOMục tiêu bao nhiêu?
→
SLACam kết với khách?

SLI

Service Level Indicator: số đo định lượng, thường 0–1. Ví dụ: request success ratio.

SLO

Service Level Objective: target nội bộ cho SLI. Ví dụ: 99.9% trong 30 ngày.

SLA

Service Level Agreement: hợp đồng với customer, có hậu quả như refund/credit.

Source: Google SRE Book — Service Level Objectives
Rule of thumb

Quan hệ đúng: SLA < SLO < current performance

Nếu SLO bằng SLA thì bạn không có vùng đệm. Miss nhẹ là thành vấn đề legal/commercial.

02
SLI Selection

Chọn SLI là chọn
cách nhìn user pain

Không phải metric nào có dashboard cũng xứng đáng làm SLI.

Good SLI tests

Một SLI tốt phải pass 3 test

1

Measurable

Có metric/log thật để tính. Không phải khái niệm mơ hồ như “user happy”.

2

User-side

Càng gần user càng tốt: load balancer, RUM, synthetic probe, client-side.

3

Proportional

SLI giảm khi user đau. User OK thì SLI không nên giảm mạnh.

Service type mapping

Mỗi loại service có SLI khác nhau

Service typeSLI nên dùngVí dụ formula
Request-response API / WebAvailability, Latencygood requests / total requests
Data processing / StreamFreshness, Correctness, Coverageevents processed within 5m lag / all events
Storage / DBDurability, Availability, Latencysuccessful read/write / all operations
Batch job / CronCompleteness, Runtimecompleted within 1h / scheduled jobs
Anti-pattern

Vì sao CPU không nên làm SLI?

CPU 80%, user vẫn OK

CPU cao có thể chỉ là service đang xử lý tốt. Nếu latency và error rate ổn, user không đau.

capacity signal

CPU 30%, user vẫn đau

Deadlock, DB lock, network timeout có thể làm p99 tăng mạnh dù CPU thấp.

not user happiness

CPU/memory/disk là saturation signal. Dùng cho capacity dashboard hoặc ticket, không phải page chính.

Latency SLI

Latency: đừng dùng average, hãy dùng percentile

p50

Trung vị. Tốt để hiểu typical user, nhưng che tail.

p95

Top 5% slowest. Hợp báo cáo general.

p99

Top 1% slowest. Default tốt cho user-facing service.

p99.9

Tier-1 critical path: login, checkout, payment.

Google SRE Book khuyến nghị percentile để thấy tail latency.

Cost ladder

04
Error Budget

Error budget là
quyền được fail có kiểm soát

Nếu SLO không phải 100%, phần còn lại chính là ngân sách lỗi.

Formula

Công thức error budget

budget = (1 - SLO) × total_events

Ví dụ: SLO = 99.9%, traffic = 1,000,000 req/day, window = 30 ngày
→ budget = 0.001 × 1,000,000 × 30 = 30,000 failed requests / month
Convert to downtime

Budget có thể quy đổi sang downtime

30k
failed requests/month
12
req/s peak
41m
downtime equivalent
downtime = allowed_failures / requests_per_second = 30,000 / 12 = 2,500s ≈ 41 phút

Policy

Error budget phải có “răng”

Nếu hết budget mà team vẫn deploy như bình thường, SLO chỉ là KPI trang trí.

Budget còn nhiều

Cho phép release nhanh hơn, thử nghiệm có kiểm soát.

Budget gần cạn

Giảm rủi ro, review rollout, ưu tiên reliability bugs.

Budget hết

Freeze release, focus reliability, postmortem các incident lớn.

05
Burn Rate

Burn rate cho biết
đang đốt budget nhanh cỡ nào

Raw error rate không đủ. Burn rate chuẩn hóa theo SLO.

Formula

Công thức burn rate

burn_rate = error_rate trong window / (1 - SLO)

Với SLO = 99.9% → allowed error rate = 0.1%
Nếu error_rate hiện tại = 5% → burn_rate = 5% / 0.1% = 50×
Normalize by SLO

Cùng 5% error nhưng mức nguy hiểm khác nhau

Service SLOAllowed errorError hiện tạiBurn rateKết luận
99%1%5%5×Đáng chú ý
99.9%0.1%5%50×Page ngay
99.99%0.01%5%500×Critical
Time to exhaust

Burn rate → bao lâu thì hết budget?

Burn rateÝ nghĩa với 30-day windowTime to exhaust
1×Đốt đúng tốc độ baseline30 ngày
2×Nhanh gấp đôi15 ngày
6×Cần page nếu kéo dài5 ngày
14.4×2% budget trong 1h~50 giờ
1440×Gần như outage toàn phần30 phút
PromQL

Burn rate trên Prometheus

# SLO 99.9% → allowed error = 0.001 ( sum(rate(http_requests_total{status=~"5..|429"}[1h])) / sum(rate(http_requests_total[1h])) ) / (1 - 0.999)

Tử số là error ratio trong window. Mẫu số là allowed error ratio theo SLO.

07
MWMBR

Multi-Window
Multi-Burn-Rate

Công thức thực chiến của SRE: giảm noise nhưng vẫn bắt outage nhanh.

Rule

MWMBR fire khi cả 2 điều kiện đúng

Long window ≥ Tsự cố đủ lớn
AND
Short window ≥ Tvẫn đang xảy ra
→
Alert firesđáng gọi người

Guideline phổ biến: short window ≈ long window / 12.

Source: Google SRE Workbook — Alerting on SLOs
Google starting table

Tier MWMBR chuẩn cho SLO 99.9%

TierLong windowShort windowBurn rateBudget burnedAction
Tier 11h5m14.42% trong 1hPage ngay
Tier 26h30m65% trong 6hPage
Tier 33d6h110% trong 3dTicket
Derive threshold

Tại sao Tier 1 là 14.4?

T = budget_fraction / window_fraction_of_SLO_period

Với 30 ngày = 720h, muốn page khi đốt 2% budget trong 1h:
T = 0.02 / (1 / 720) = 14.4

Nghĩa là: nếu tiếp tục cháy ở tốc độ này, service sẽ hết budget trong ~50 giờ.

Prometheus rule

Rule Tier 1 bằng PromQL

groups: - name: slo-alerts rules: - alert: SLO_BurnRate_Tier1 expr: | ( sum(rate(http_requests_total{status=~"5..|429"}[1h])) / sum(rate(http_requests_total[1h])) ) / (1 - 0.999) >= 14.4 AND ( sum(rate(http_requests_total{status=~"5..|429"}[5m])) / sum(rate(http_requests_total[5m])) ) / (1 - 0.999) >= 14.4 labels: severity: page tier: "1" annotations: summary: "Burn rate ≥ 14.4 over 1h and 5m" runbook: "https://runbooks.example.com/slo-burn"
Recovery behavior

Vì sao cần AND?

Nếu chỉ dùng long window

Incident hết rồi nhưng 1h window vẫn cao → alert dính lâu, spam on-call.

Long AND short

Short window về dưới threshold sau ~5 phút → alert stop nhanh dù long window còn cao.

Prometheus semantics

Alert rule không chỉ có expr

expr

PromQL điều kiện alert. Vector element thỏa điều kiện sẽ active.

for

Yêu cầu điều kiện tồn tại trong một duration trước khi firing. Giảm spike noise.

keep_firing_for

Giữ alert firing thêm một thời gian sau khi condition biến mất, giảm flapping.

Source: Prometheus alerting rules docs

Quick check

Câu hỏi kiểm tra 1

Câu hỏi: Vì sao fingerprint/SLI không nên include timestamp?
Đáp án: Timestamp thay đổi ở mọi event. Nếu include timestamp, hai request/alert cùng bản chất sẽ thành hai nhóm khác nhau; hệ thống không aggregate được, không tính ratio/window đúng.
Quick check

Câu hỏi kiểm tra 2

Câu hỏi: SLO 99.9%, error rate hiện tại 2%. Burn rate là bao nhiêu?
Đáp án: Allowed error = 1 - 0.999 = 0.001 = 0.1%. Burn rate = 2% / 0.1% = 20×.
Quick check

Câu hỏi kiểm tra 3

Câu hỏi: Vì sao MWMBR dùng long window AND short window?
Đáp án: Long window xác nhận sự cố đủ lớn để đáng action. Short window xác nhận sự cố vẫn đang xảy ra. AND giúp giảm false positive và giúp alert tắt nhanh sau khi sự cố hết.
Quick check

Câu hỏi kiểm tra 4

Câu hỏi: 4xx có nên đếm vào fail SLI không?
Đáp án: Thường không, vì 4xx đa số là client/user error. Nhưng 429 nên đếm là fail vì system đã reject user do rate limit/capacity.

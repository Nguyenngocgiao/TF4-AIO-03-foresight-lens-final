SLO, Error Budget,
Burn-Rate Alerting

Define what is worth paging on before building detectors — reduce pager noise, raise signal-to-noise ratio, bind every metric to user experience.

The problem

An accurate detector ≠ a useful alert

3σ on CPU, IsolationForest on latency — all tests pass. Deploy. First week on-call:

23

alerts/week — pager firing

19

transient spike, self-recovered

1

real incident — alert fatigue caused 12-minute response delay

Anomaly ≠ user impact. A CPU spike doesn't mean users are affected. A 50ms latency increase from 100 → 150ms doesn't either. The pager should fire when users actually see errors or failed requests — and only then.
01
Definitions

SLI · SLO · SLA

3 confusable letters — get the distinction wrong, design wrong

LetterWhat it isFor whomPenalty on miss
SLI IndicatorA measurement, range 0-1, from the user's side, proportional to user painEngineers—
SLO ObjectiveTarget for the SLI, e.g. 99.9% over a 30-day windowEngineering teamRelease freeze, reliability work
SLA AgreementContractual commitment to the customerCustomerMoney (refund, credit)
Relationship: SLA < SLO < current. SLA is looser than SLO; SLO is looser than the current measurement — leaving buffer to fix issues before the SLA is missed.

SLI selection — 4 service types, 4 standard metrics

Google SRE Workbook Chapter 2 — pick the right metric for each system type.

Service typeSLI categoriesExample formula
Request-response (API, web)Availability, Latencycount(2xx,3xx,4xx_not_429 AND latency<200ms) / count(all)
Data processing (ETL, stream)Freshness, Correctness, Throughputcount(events processed within 5min lag) / count(all)
Storage (S3-like, DB)Durability, Availabilitycount(read OR write success) / count(all)
Batch job (cron)Completeness, Runtimecount(completed within 1h) / count(scheduled)

Anti-pattern: CPU as an SLI

The most common mistake. CPU is not proportional to user pain — two illustrative cases:

False positive

CPU 80%, latency 150ms

Users still happy. But the SLI says "not OK" → pages on-call at 2am. Pager noise.

False negative

CPU 30%, deadlock, latency 5s

Users suffer. SLI says "OK". Silent failure — no alert, users can't be served.

CPU is a saturation signal, not a user happiness metric. Keep it on the capacity dashboard, not as an SLI.

Latency: which percentile?

PercentileCapturesUse when
p50 (median)Median — ignores the tailThroughput-bound systems (data pipelines)
p95Top 5% slowestGeneral reporting
p99Top 1% — tail captures real user painDefault for user-facing services
p99.9Top 0.1%Tier-1 services (search, login)

Why not p50? If 50% of requests are fast and 50% take 10s, p50 stays fast → SLI reports OK while half the users are suffering.

Availability: which status codes count as failure?

HTTP rangeCount as fail?Reason
5xxYESServer error — clearly our fault
429YESRate-limited by the system → system is rejecting users
4xx (excluding 429)NOUser-side error (bad request)
2xx, 3xxNOSuccess
Count 4xx as failure and the SLI gets dragged down by bots and scrapers. Exclude 4xx (except 429) → fair signal.

The cost per nine

Set SLO 99.999% for every service = miss it in the first month = the SLO gets ignored. Pick close to current performance, stretch one tier.

SLOArchitecture required
99%Single instance, manual recovery
99.9%Multi-instance + LB + auto-failover
99.99%Multi-AZ + automated runbook + 24/7 on-call
99.999%Multi-region active-active + dedicated SRE team
Rule of thumb: core user-facing services (login, checkout) → 99.9%-99.95%. Internal services (admin, analytics) → 99%. Background batch → SLO on freshness, not availability.

Error budget — how much you're allowed to break

budget = (1 − SLO) × total_events

Count-based example

SLO 99.9%, traffic 1M req/day, 30-day window:

budget = 0.001 × 1M × 30 = 30,000 fails/month

Convert to downtime

If an incident causes 100% failure at 12 req/s peak:

30,000 / 12 = 2,500s ≈ 41 minutes

03
Burn Rate

Rate of budget consumption

Burn rate — normalize error_rate by SLO

A raw "5% error rate" doesn't tell you whether to page. It depends on the SLO:

burn_rate = error_rate / (1 − SLO)

SLO 99% → 5% error

burn_rate = 5 → moderate, no page needed

SLO 99.9% → 5% error

burn_rate = 50 → page immediately

Burn rate gives a single threshold usable for any service, regardless of SLO target.

Burn rate → time to exhaust the budget

SLO 99.9%, 30-day window:

MWMBR

Multi-Window Multi-Burn-Rate

Why a single-window alert isn't enough

An unavoidable trade-off if you stick to one window:

WindowFalse positiveMTTDStop-firing lag
1 minuteHigh1-2 min<1 min
5 minutesMedium3-5 min5 min
1 hourLow5-15 min1 hour — spam
6 hoursVery low30+ min6 hours — long spam
Short window = noisy. Long window = slow + sticks around after the incident ends. Solution: use two windows together.

MWMBR — alert fires when BOTH conditions are true

Google SRE Workbook Chapter 5, Section 6. The original formula:

Long window

burn rate ≥ T → "the incident is large enough to warrant action"

Short window AND

burn rate ≥ T → "the incident is still happening right now"

Ratio rule: short ≈ long / 12. Drop the short window → the alert sticks 1-6h after the incident ends. AND is the key.

Standard tier table — Google SRE

TierLongShortBurn rate ≥Budget burnedAction
1 page1h5min14.42% in 1hPage on-call immediately
2 page6h30min65% in 6hPage on-call
3 ticket3d6h110% in 3dBusiness hours

Threshold derivation: T = budget_fraction / (window / SLO_period). Example Tier 1: want 2% burned in 1h → T = 0.02 / (1h/720h) = 14.4.

Prometheus alert rule — Tier 1 example

Both the 1h window and the 5min window must exceed threshold 14.4:

- alert: SLO_BurnRate_Tier1
  expr: |
    (
      sum(rate(http_requests_total{status=~"5..|429"}[1h]))
      / sum(rate(http_requests_total[1h]))
    ) / (1 - 0.999) >= 14.4
    AND
    (
      sum(rate(http_requests_total{status=~"5..|429"}[5m]))
      / sum(rate(http_requests_total[5m]))
    ) / (1 - 0.999) >= 14.4
  labels: {severity: page, tier: "1"}
  annotations:
    summary: "Burn rate >= 14.4 over both 1h and 5m"
    runbook: "https://runbooks.example.com/slo-burn"

Verify syntax: promtool check rules burn_rate_alerts.yaml

Recovery behavior — why MWMBR doesn't spam

When the incident ends, the alert clears quickly thanks to the short window:

After incident endsShort window (5min)Long window (1h)AND?Alert state
0 minhighhightruefiring
5 minbelow thresholdhighfalseSTOP
30 minlowstill highfalsestopped
1 hourlowlowfalsestopped
The alert clears after 5 minutes, not 1 hour. The short window makes alerts recover fast — no noise once the incident is over.

After the alert fires

Closed-loop auto-remediation

MWMBR fires → pages on-call → human opens runbook → types restart command. Most Tier-1 pager events in production have a deterministic runbook (restart, scale up, fail over). Auto-remediation lets the machine execute those runbooks itself, freeing humans for incidents that genuinely need judgment.

Tier-based decision matrix

SymptomAuto-action
Pod 5xx > 50% over 2 minkubectl rollout restart
Queue depth > 10kScale workers +50%
Replica lag > 60sFail read traffic → replica2
Spike, root cause unknownNO AUTO — page

Rule: an action goes auto only when rollback is safe AND the symptom has been seen many times with the same fix.

Observe → Decide → Act loop

  1. Observe — burn-rate alert fires (Tier-1, condition holds ≥ 2 min)
  2. Decide — match symptom to runbook catalog, check preconditions (not in deploy window, blast radius OK)
  3. Act — execute via API/CLI (kubectl, AWS SSM, custom)
  4. Verify — wait the verification window, check probe + metric. Not recovered → rollback + page
  5. Log — emit audit event (who, what, outcome, time-to-recover)
runbooks:
  - match:
      alert: SLO_BurnRate_Tier1
      service: api-gateway
    preconditions:
      - not_in_deploy_window
      - blast_radius_ok        # <= 1 pod restart per 5 min
    action: kubectl rollout restart deployment/api-gateway
    verify:
      probe: http://probe/api/healthz
      pass_rate_min: 0.95
      window_seconds: 90
    on_fail: page              # rollback gate → human
    audit: slack#incident-log
The rollback gate is the safety net. Without verify + rollback, auto-remediation becomes auto-destruction. Probe pass-rate in the verify window decides pass/fail. When in doubt → page a human, don't act. Tools: AWS Systems Manager Runbooks, PagerDuty Rundeck, Argo Rollouts, Keptn.

Chaos Engineering

Definition

What chaos engineering is

An empirical discipline that deliberately injects faults into a distributed system to surface weaknesses before they occur in production. Unlike unit tests (verify code matches spec), load tests (verify capacity), and pen tests (find security holes) — chaos engineering targets reliability weaknesses arising from component interaction, a class of bugs none of those methods catch.

PracticeGoalWhen run
Unit testVerify code matches specCI/CD on every commit
Load testVerify expected capacityPre-launch, periodic
Penetration testFind security vulnerabilitiesQuarterly
Chaos engineeringFind reliability weaknesses from component interactionContinuously in production-like environments
5 core principles (principlesofchaos.org): (1) Build a hypothesis around steady-state behavior. (2) Vary real-world events. (3) Run experiments in production. (4) Automate experiments to run continuously. (5) Minimize blast radius.

Sources: principlesofchaos.org (2017, refined 2019); Rosenthal & Jones, Chaos Engineering: System Resiliency in Practice, O'Reilly 2020.

Why chaos — reason 1

Sample-size problem + cost asymmetry

Production incidents are rare events. An average service sees 1-3 outages per quarter — sample size far too small to verify the alerting + RCA pipeline actually works. Waiting for real outages to learn whether the alert catches them = every outage is one expensive lesson for the team and users. Chaos shifts reliability from "react when broken" to "verify continuously on a known-good baseline."

Without chaos

~12 samples/year

~3 prod incidents/quarter → 12 data points/year to validate the alerting pipeline. Detector gaps don't surface until a real outage hits, typically outside business hours. Each outage = engineering cost + user impact + reputation damage.

With chaos

~520 samples/year

10 experiments/week × 52 weeks. Each experiment tests one specific alert path. Detector gaps surface during business hours, leaving time to fix the pattern before it appears in real production. 43× more validation signal.

Cost asymmetry

~$5,600/min vs ~$15/hr

Industry average enterprise downtime cost (Gartner): ~$5,600/minute. Cloud chaos infrastructure cost: ~$5-15/hour (compute + observability). Break-even after the first prevented incident.

Sources: Google SRE Workbook ch.5; Gartner IT Downtime Cost survey (widely cited industry benchmark); Basiri et al., "Chaos Engineering", IEEE Software 2016.

Experiment design — steady-state hypothesis + 5 fields

An experiment without a hypothesis is vandalism. A well-designed experiment defines exactly what "OK" looks like before injection, predicts what should happen during injection, and identifies which metric will detect a mismatch. The 5-field framework below (Rosenthal & Jones, O'Reilly 2020 ch.4) is the baseline for any serious experiment.

FieldWhat it isConcrete exampleWhen to apply
Hypothesis Testable claim about steady-state under fault "Order success ≥ 99.5%, p99 ≤ 800ms when payment-svc is partitioned from checkout-svc for 60s, because retry logic fails over to backup payment provider within 30s" Can't write a one-sentence prediction → skip the experiment
Blast radius Explicit damage limit "1 pod, 10% traffic, 60s duration, 1 region (canary cell)" Start at the smallest meaningful scale, escalate through 5 stages only after the prior stage passes
Rollback Auto-abort condition + cleanup command "Abort if probe pass-rate < 50% for 20s. Cleanup: tc qdisc del dev eth0 root" Rollback must work even if everything else fails — test rollback before the experiment
Measurement Concrete metrics + capture window probe_pass_rate, p99_latency, error_rate, rca_root_service; capture t-5min → t+10min Decide metrics before injection — post-hoc selection = confirmation bias
Ground truth Expected pipeline behavior "Detection fires < 30s. RCA picks payment-svc (not checkout-svc — that's the symptom carrier)" Without ground truth, you can't compute precision/recall

Source: Rosenthal et al., Principles of Chaos Engineering, principlesofchaos.org (2017, refined 2019); Rosenthal & Jones, O'Reilly 2020 ch.4.

Experiment lifecycle — 5 stages with a rollback gate

After designing the 5 fields, the runtime executes the experiment as a 5-stage loop. Stage 1 confirms steady-state before injection — if the precondition fails, abort before causing damage. Stage 4 triggers rollback automatically when the abort condition is hit. Stage 5 scores and loops to the next experiment after a 120s cooldown.

Key insight: the 120s cooldown between experiments lets the system return to baseline (metrics must match the baseline.json captured before the first experiment). Skip the cooldown → results contaminate from the prior experiment, and you can't distinguish "the pipeline caught fault N" from "the pipeline hadn't recovered from fault N-1".

Source: Rosenthal & Jones, Chaos Engineering, O'Reilly 2020 ch.4 (experiment lifecycle); Netflix Tech Blog "Automating Failure Testing Research at Internet Scale" (2016).

Outcome metrics — chaos scoreboard

Every experiment produces one labeled data point. After N experiments, the chaos pipeline has a TP / FP / FN / TN matrix — you can compute precision, recall, and MTTD for alerting + RCA. This is how you convert "intuition about reliability" into concrete pass/fail metrics you can track over time.

MetricDefinitionProduction-chaos target
Detection recallDetected experiments / total experiments≥ 80%
RCA accuracyRCA picked correct service / detected experiments≥ 70%
MTTD p50Median seconds from inject to alert fire< 30s
False alarm rateAlerts in baseline (no-fault) windows / total baseline windows< 5%
Example 10-run interpretation: detected 8/10, RCA correct 6/8, MTTD p50 28s, false alarms 1/20 baseline windows → recall 80%, RCA 75%, FAR 5%. Detection passes threshold but RCA is borderline → investigate the 2 RCA failures before broadening experiment scope or scaling traffic. Do not tune the pipeline to force a pass — that is confirmation bias, and the result will not generalize to real production variation.

Source: Basiri et al., "Chaos Engineering", IEEE Software 2016 (Netflix ChAP measurement framework); Adrian Cockcroft, "Failure Modes in Microservices", AWS re:Invent 2019.

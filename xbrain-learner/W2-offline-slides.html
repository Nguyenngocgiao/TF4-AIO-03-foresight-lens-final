Culprit vs Victim

The culprit rarely shouts loudest

The service with the most alerts is usually not the root cause.

Counter-intuitive

checkout-svc has 12 alerts. payment-svc has 14. Root cause is payment.

Alert count looks like the obvious signal — but it embeds the cascade direction inversely. Surface-area services bubble up many derivative alerts; the actual culprit deep in the chain often has just a handful.

Alert count per service in cluster: payment-svc ████████████████ 14 checkout-svc ████████████ 12 edge-lb █████████ 9 notification-svc ██████ 6 If you pick "most alerts" → correct here. But in many other incidents: api-gateway ███████████████ 15 ← actually victim auth-svc ████ 4 ← actually culprit Alert count is not reliable.

Why the culprit doesn't always alert the loudest

When a service fails, it pushes errors to every caller. Each caller has many metrics → each caller produces many alerts. Services deep in the dependency chain often have fewer surface metrics (just their output), while services at the boundary have larger surface areas and noisier monitoring.

Three real signals of a culprit

  • Earliest alert timestamp in the cluster
  • Downstream-most position on the service graph
  • Alerts are internal (CPU, connection pool, GC) rather than boundary (latency, 5xx)

Topology

What a graph edge really is

Before trusting any graph-based RCA, understand where the graph itself comes from.

Edge anatomy

One edge = thousands of aggregated calls

A graph edge is not abstract. Thousands of individual call spans within a time window collapse into a single directed relationship between two services, carrying aggregate metrics.

One raw span

trace_id=3ccadf44… · parent=checkout-svc · target=payment-svc · op=POST /charge · duration=820 ms · status=OK

One aggregated edge

checkout → payment · window=5 min · count=4217 · error_rate=0.02 · p99=910 ms · last_seen=2s ago

Sources

Three sources of topology truth — each with its blind spot

No single source covers every dependency. Real stacks combine two or three layers.

Production pattern: tracing as primary coverage (~80%) + mesh or eBPF to fill async and direct-connection gaps. IaC overlays a "should-be" graph to detect drift between intended and actual topology.

03 — TopologyW2 RCA
Direction

Call direction vs blame direction

The graph carries one set of edges. RCA reads them in different directions depending on the question.

RCA insight: alerts trip in call order (edge-lb first, payment last) but the root sits at the blame origin — the rightmost service. Walking the call graph in reverse lets PageRank accumulate blame at the deep service.

Bidirectional edges: some services call each other (datapower ↔ esb, billing ↔ user-service). In those cycles, blame can flow either direction and PageRank converges ambiguously. Granger causality on the underlying metrics resolves the tie.

03 — TopologyW2 RCA
Fidelity issues

Where the topology gets it wrong — four common failure modes

Production graphs drift, miss edges, and lag reality. Each failure mode below skews downstream RCA in a specific way.

Missing async edges

Kafka, SQS, RabbitMQ traffic rarely shows in trace data. Producer and consumer look disconnected. When the consumer is the actual culprit, PageRank ranks it last — no incoming edges.

Drift after deploy

A service deployed 5 minutes ago hasn't accrued enough calls to register edges. Or its name was changed and the graph still lists the old identity. SREs lose confidence when output doesn't match production reality.

External edges

Calls to Stripe, Twilio, partner APIs, external DB clusters are the edges of your observed universe. When an external dependency is the root cause, RCA points at your caller. Half-right; needs human investigation.

Multi-tenant / shared services

A single database serves 20 microservices. PageRank flags the DB correctly, but the actionable insight ("noisy tenant X") needs per-tenant labels on calls — an extra layer beyond the structural graph.

Mitigation: pair topology RCA with metric drift and causal inference. When the graph fails, time-based signals (which metric drifted first) give an independent path to root cause that needs no graph at all.

Graph RCA

Why we trust PageRank more than count

Graph score

Reverse-graph PageRank — "who do many callers depend on"

Run PageRank on the reverse of the service call graph. Mass accumulates at services that are heavily depended on by alerting peers — exactly where the culprit usually lives.

Service subgraph (alerting only): edge-lb │ ▼ checkout-svc │ ┌─────┼─────┐ ▼ ▼ ▼ payment cart notification -svc -svc -svc Reverse graph (PageRank input): payment-svc ← checkout-svc ← edge-lb cart-svc ← checkout-svc notification-svc ← checkout-svc PageRank "vote" flow: callers vote for callees. payment-svc receives the most votes.

Output

ServicePageRankRank
payment-svc0.421
checkout-svc0.282
edge-lb0.153
notification-svc0.084
cart-svc0.075

Confidence: 0.42 / (0.42+0.28+0.15+0.08+0.07) = 0.42 → "moderately confident".

Caveat: PageRank uses topology only. If the service graph is missing an edge, the score is skewed. This is why we add timestamp ranking and LLM enrichment as additional signals.

04 — Graph RCAW2 RCA
Causal inference

When two candidates tie on graph score

PageRank ranks by topology structure. When the top-1 and top-2 candidates have similar scores, you need a second, time-based signal — Granger causality on their metrics.

What Granger causality tests

Whether past values of metric A help predict future values of metric B, beyond what B's own past already explains.

Input: two aligned timeseries (e.g., payment.db_pool_used and checkout.latency_p99), plus a candidate lag window.

Output: a p-value indicating whether A Granger-causes B, plus the optimal lag in seconds.

Not "true" causality — it's predictive precedence. But for ranked RCA, predictive precedence is what you need.

Worked example on the cluster

Pairp-valueLag
payment.db_pool → checkout.latency0.00315 s
checkout.latency → payment.db_pool0.41—
edge-lb.5xx → checkout.latency0.1830 s

payment.db_pool Granger-causes checkout.latency with high confidence (p<0.01). The reverse direction is not significant. This is independent evidence for "payment is the culprit".

When to use: top candidates have PR scores within 10% of each other, or topology is known to be incomplete. When to skip: the gap between candidates is wide; Granger adds little.

04 — Graph RCAW2 RCA
Fusion

Combining N rankers — Weighted Reciprocal Rank Fusion

Production stacks rarely rely on a single ranker. RRF combines several independently-derived rankings into one robust score. Used by BARO, NSigma, TopoWalk, Onset — every mainstream RCA framework.

Why RRF, not score averaging

Different rankers produce scores on different scales — PageRank is 0..1, drift-count is 0..N, causal is 0..p. Averaging raw scores lets the highest-magnitude ranker dominate.

RRF uses ranks, not scores. Each service gets 1/(k+rank) from each ranker, summed with per-ranker weights. Scale-invariant by construction.

k=60 is a smoothing constant that prevents rank-1 from dominating. Larger k flattens the curve.

Worked example, 3 rankers

ServicePRDriftCausalFused
payment-svc1210.030
checkout-svc2130.028
edge-lb3320.025

Weights = [0.4, 0.4, 0.2]. Even though checkout-svc ranks #1 on drift, payment-svc wins because it's strong across all three rankers.

Confidence from fused score: the gap between top-1 and top-2 matters more than the absolute score. Gap > 30% → high confidence. Gap < 10% → page a human, don't auto-act.

05
LLM augmentation

When does an LLM earn its latency

And when does it only add latency and hallucination.

Where LLMs help

The LLM does not pick the root cause. It does three other things.

1

Classify the failure kind

The graph says which service. The LLM says what kind of failure: connection_pool_exhaustion, slow_query, memory_leak, rebalance_storm.

Why it matters: the failure kind decides the action. Pool exhaustion → restart + scale. Slow query → kill + tune index.

2

Suggest an action

A concrete action: "Rollback payment-svc v3.2 to v3.1". Drawn from remediations of similar past incidents.

Why it matters: SREs don't have to remember. Recurring patterns have familiar remedies.

3

Natural-language reasoning

Two or three sentences explaining why service X is the root cause, displayed for the on-call to read at a glance.

Why it matters: helps on-call decide "trust or don't trust" the output within 30 seconds.

The LLM does not replace graph RCA. It translates the graph output into those three artifacts.

05 — LLM augmentationW2 RCA
Retrieval

Fetching similar past incidents — two approaches

Before calling the LLM, retrieve the 3 historical incidents most similar to the current cluster. Two approaches exist; choose by history size.

Heuristic match — small history

How it works: score by explicit overlap — matching services, matching severity, root-cause service appearing in the current cluster.

Input: current cluster (services list, severity) + historical incidents with structured fields.

Output: top-3 incidents with scores in 0..1.

Use when: history under ~100 incidents, service names are stable, structured fields are clean.

Weak when: services get renamed ("payment-svc" → "billing-svc"), or the same pattern appears with different wording ("DB pool" vs "connection pool").

Semantic embedding — large history

How it works: encode each incident into a vector from summary + services list. Encode the query the same way. Find top-K by cosine similarity.

Input: same as heuristic, plus free-text summaries.

Output: top-3 incidents with cosine scores 0..1.

Use when: history ≥ 500 incidents, frequent service renames, or when semantic matching matters more than structural overlap.

Cost: embedding model (~80 MB for MiniLM) and pre-computed vectors of history.

Rule of thumb: start heuristic. If it misses too much (precision under 60% on labeled history), upgrade to embeddings. Don't jump straight to embeddings while history is small — overhead exceeds benefit.

05 — LLM augmentationW2 RCA
Production concern

LLMs hallucinate — three defense layers

Layer 1

Prompt instruction

"Pick root_cause only from this services list. Pick class only from this enum: [...]. If unsure, return class='other'."

Effective but not enough — LLMs still slip 5–10% of cases.

Layer 2

Output validation

Parse JSON. Check root_cause appears in the services list. Check class is in the enum. Check confidence is in [0,1].

Hand-coded checks, don't trust the LLM blindly. This is the hardest layer.

Layer 3

Fallback

If Layer 2 fails → return graph top-1, class='other', action=['Investigate manually']. The endpoint never crashes.

Being honest about uncertainty beats giving a wrong action confidently.

Self-check

When the LLM hallucinates in real traffic (returns a service not in the list, or a class not in the enum), which layer catches it first? Which layer is the last resort?

07
Pipeline as product

Four layers, one endpoint

Correlator + RCA + LLM enrichment + platform serving — composed into a single endpoint that ingests raw incidents and returns root cause with recommended actions.

Architecture

Four layers, each with one responsibility

Production pipelines are rarely monoliths. Splitting into layers makes it possible to test each independently and to swap a failing one without rebuilding the whole pipeline.

Correlator (L1)

Module correlate.py. Receives a raw alert stream, groups it into meaningful clusters using fingerprint dedup, time-window grouping, and topology-aware collapse.

Sensitive parameters: gap_sec for time-window, max_hop for topology subgraph. Too small → cascades split into many incidents. Too large → unrelated traffic merges in.

RCA Engineer (L2)

Module rca.py. Takes a cluster, returns a ranked list of likely root-cause services. Combines graph centrality (reverse-topology PageRank) with earliest-drift timestamps.

Failure mode: a missing edge in the graph skews PageRank. When top-1 and top-2 are close (score gap under 10%), confidence is low — don't auto-act.

LLM Integration (L3)

Module enrich.py. Takes the L2 top root cause, retrieves similar past incidents, builds a prompt, calls the LLM for recommended action. A guardrail filters hallucinations.

Failure mode: LLM returns 429 (rate limit) or times out. Needs retry with backoff and a template-based fallback when the LLM is down. Per-call cost should be tracked.

Platform (L4)

Modules serve.py + pipeline.py. FastAPI endpoint accepts POST /incident, validates schema, chains L1 → L2 → L3, formats the response. /healthz for liveness, /readyz for readiness.

End-to-end latency budget ≤ 3 s. Concurrent requests need queueing or async — never block. Schema validation is strict — wrong shape returns 400 with details.

07 — ArchitectureW2 RCA
Failure modes

Five places an RCA pipeline tends to break

These are not theoretical concerns — they show up the moment the pipeline runs on a different machine or under real load instead of a sample dataset.

1. Tight layer coupling

When the correlator changes its output schema, the RCA module breaks. When RCA changes its ranking format, the LLM module breaks. Inter-layer contracts must be pinned with pydantic models or JSON schemas, versioned explicitly.

2. Hardcoded paths / envs

Pipeline runs on the author's machine but fails when deployed. Usual suspects: absolute paths, env vars set in shell rc instead of .env, dependencies missing from requirements.txt.

3. Evidence detached from code

The code runs correctly but lacks a decision log. Two weeks later, no one remembers why gap_sec=120 was chosen. Each significant parameter needs its rationale captured at the source — in an ARCHITECTURE.md or a notable inline comment.

4. Mocking the LLM instead of handling 429

A common shortcut: replace the LLM with a canned response so demos don't hit rate limits. But production will hit real 429s. The proper fix: retry with exponential backoff, fall back to template-based responses, and surface the fallback in the response metadata.

5. Single-instance assumptions

The pipeline assumes one worker — keeping in-memory state (dedup cache, cluster buffer). Scaling to 3 replicas fragments the state; dedup stops working. Stateless design or an external state store (Redis) needs to be planned upfront.

Verify before committing

Run alerts_sample.jsonl through the endpoint from a clean checkout (clone fresh, pip install, run). Catch schema errors there — much cheaper than debugging them at full scale.
